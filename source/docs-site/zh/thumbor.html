Thumbor 兼容

Thumbor 风格路径的行为与 AWS 方案完全一致——本页是迁移存量 CloudFront 图片 URL 团队的重点参考。

路径结构

/[fit-in/][AxB:CxD/][WxH/][filters:f1(v):f2(v)/]key
段含义示例
fit-in以 fit=inside 缩放(留白而不是裁剪)。/fit-in/300x300/cat.jpg
AxB:CxD手工裁剪:left=A、top=B、width=C−A、height=D−B(各 1–6 位数字)。/10x20:110x220/cat.jpg
WxH目标尺寸。某一维为 0 表示自适应,并隐含 fit=inside。/300x0/cat.jpg
filters:…冒号分隔的 filter 列表。应用前会先按字母序排序(保证 format 一定先于 quality)。/filters:format(webp):quality(80)/cat.jpg
key默认源桶中的对象 key,经 URL 解码。photos/cat.jpg

桶前缀:s3: 与 gs:

路径中形如 s3:<bucket>/ 或 gs:<bucket>/ 的段可指定其他源桶。该桶在 SOURCE_BUCKETS 白名单内则覆盖默认桶,否则该段被忽略。

/300x300/gs:my-other-bucket/photos/cat.jpg
/300x300/s3:legacy-aws-bucket/photos/cat.jpg   ← 迁移后仍然接受

BUCKET_MAP:不改 URL 迁移 S3 地址

如果存量 URL 里内嵌的 S3 桶名在 GCS 上并不存在,可用 BUCKET_MAP 配置逗号分隔的 s3名=gcs名 映射:

BUCKET_MAP="prod-images-us-east-1=my-gcs-images,assets-bucket=my-gcs-assets"

此后 /300x300/s3:prod-images-us-east-1/cat.jpg 会透明地读取 gs://my-gcs-images/cat.jpg。白名单校验针对映射后的 GCS 桶名。再加上完全一致的签名算法,大多数 AWS 客户端只需改一条 DNS 即完成迁移。

filter 按字母序而非 URL 书写顺序应用,与 AWS 实现一致。这保证了无论 URL 怎么写,format() 都先于 quality() 执行。

自定义重写(Custom 请求)

若存量 URL 不符合 Thumbor 语法,可配置 REWRITE_MATCH_PATTERN (正则)与 REWRITE_SUBSTITUTION。命中的路径先重写、再按 Thumbor 解析——与 AWS 方案的 Custom request 机制相同。

REWRITE_MATCH_PATTERN="/thumbs/(\d+)/(.*)"
REWRITE_SUBSTITUTION="/fit-in/$1x$1/$2"
# /thumbs/200/cat.jpg  →  /fit-in/200x200/cat.jpg

迁移清单(CloudFront → Cloud CDN)

  1. 部署时把 SOURCE_BUCKETS 指向 GCS 桶;URL 中内嵌的 S3 桶名逐一加入 BUCKET_MAP。
  2. 把签名密钥 JSON 原样复制进 Secret Manager,SECRET_KEY 保持不变——存量签名 URL 直接可用。
  3. 沿用 AWS 部署中的 REWRITE_MATCH_PATTERN/REWRITE_SUBSTITUTION 取值。
  4. 把图片域名的 DNS 指向负载均衡器 IP(建议先用预发域名验证)。
  5. 对新端点跑 deployment/run-e2e-tests.sh,并抽样对比生产 URL 的输出(逐字节或目视)。
