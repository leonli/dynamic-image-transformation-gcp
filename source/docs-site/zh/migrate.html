从 AWS 迁移

本页面向已在生产环境使用 AWS 解决方案 Dynamic Image Transformation for Amazon CloudFront(v7 serverless) 的团队,帮助你在客户端零改动的前提下迁移到本方案。 URL API、签名算法、错误契约与 AWS 逐字段一致(见 兼容性规格说明)—— 你的应用、移动端、CMS 模板、已签发的签名 URL 全部原样可用,唯一变化的只是 DNS 记录指向哪里。

本指南的完整版本——包含 S3 日志回放脚本和每一条 CLI 命令、可直接复制 执行——在代码仓库中: docs/MIGRATION_zh.md。 本页汇总迁移方案、步骤与最佳实践。

迁移全景

迁移全程两套栈并行运行,没有"一刀切"时刻,也不需要停机窗口。流量靠加权 DNS 逐步转移;图片通过增量、可重复执行的传输任务从 S3 流向 GCS。

阶段做什么客户端影响典型耗时
1. 评估盘点 AWS 栈:参数、桶、流量、实际用到的功能无1–2 天
2. 迁移图片S3 → GCS,用 Storage Transfer Service(增量、可重复)无数小时–数天
3. 迁移签名密钥同一把 HMAC 密钥进 Secret Manager → 旧签名 URL 继续有效无几分钟
4. 部署 GCP 栈Launch Wizard 或 Terraform,参数 1:1 对应无约 30 分钟
5. 验证e2e 套件 + 回放真实生产 URL无1–2 天
6. 预签发 TLS 证书Certificate Manager DNS 授权——切流之前证书就 ACTIVE无约 1 小时
7. 切流加权 DNS 10% → 50% → 100%,低 TTL,盯错误率无感知1–7 天
8. 回滚(如需)DNS 权重拨回去——AWS 栈一直在运行无感知分钟级
9. 下线 AWS删除 CloudFormation 栈;S3 保留一个观察期后再清无观察期后

第 1 步 — 评估现有 AWS 部署

拉取 CloudFormation 参数(aws cloudformation describe-stacks)和 图片处理 Lambda 的环境变量——它们与本方案的 Terraform 变量 1:1 对应。再从一天的 CloudFront 访问日志里确认生产实际用了哪些请求类型:base64 开头的路径 (/eyJi…)是 Default,/fit-in/… 和 /filters:… 是 Thumbor,其余是 Custom。

第一天就做:把图片域名的 DNS TTL 降到 60 秒。TTL 变更要等 一个旧 TTL 周期才全网生效,提前做,等于免费拿到秒级切流和秒级回滚。

BUCKET_MAP 是迁移的关键辅助。如果生产 URL 内嵌了 S3 桶名(Default 请求 JSON 里的 "bucket" 字段,或形如 /my-s3-bucket:photos/cat.jpg 的路径),你不需要重新生成任何 URL。 配置 bucket_map = "my-s3-bucket=my-gcs-bucket",所有指名该 S3 桶的请求会被透明地转到对应 GCS 桶。显式 s3: 前缀同样接受并映射; SOURCE_BUCKETS 白名单校验作用在映射后的桶名上,与 AWS 语义一致。

第 3 步 — 复制图片(S3 → GCS)

推荐 Storage Transfer Service:全托管、并行、带校验和,最重要的是增量、可重复 执行。先跑一次全量,之后重跑(或调度),把切流前写入 S3 的新图持续同步:

gcloud transfer jobs create s3://my-s3-bucket gs://my-gcs-bucket \
    --source-creds-file=/tmp/aws-creds.json --name=dit-migration-images
gcloud transfer jobs run dit-migration-images   # 随时增量重跑

对象 key 必须保持一致。方案按 URL/JSON 中的精确 key 取对象 ——复制过程中不要改名、加前缀或拍平目录。若 ENABLE_DEFAULT_FALLBACK_IMAGE=Yes,兜底图对象也要一并复制。

第 4 步 — 迁移签名密钥

签名是 HMAC-SHA256(path[?sorted-query]) 的 hex 摘要——同一把密钥 在两朵云上算出同样的签名。把密钥迁过来,所有已签发的签名 URL(包括嵌在邮件、 App 里的长期链接)继续有效:

aws secretsmanager get-secret-value --secret-id <secret> \
    --query SecretString --output text > /tmp/dit-sig.json

terraform apply -var-file=terraform.tfvars \
    -var "signature_secret_json=$(cat /tmp/dit-sig.json)"

secret_key_name 保持与 AWS 的 SecretsManagerKeyParameter 一致。完成后删除临时文件,永远不要提交 到代码库。

第 5 步 — 部署与验证

用 Launch Wizard 或 Terraform 部署,参数值照抄 AWS。然后在任何流量切换之前完成验证:

  • 跑内置套件:deployment/run-unit-tests.sh 和 BASE_URL=http://<lb_ip> deployment/run-e2e-tests.sh。
  • 回放真实生产 URL——从 CloudFront 访问日志提取 top-N 路径, 对两套栈逐一请求,对比状态码、错误 JSON 和 Content-Type (必须完全一致)。字节大小因编码器版本可能略有差异,比像素不比字节。 smartCrop 裁剪等价但非字节一致(Vision 与 Rekognition 的人脸框略有不同)。
  • 验证用现有 AWS 签名代码生成的签名 URL 在 GCP 侧验签通过,篡改后 返回 403 SignatureDoesNotMatch。
  • 用两次 GET 测 CDN 缓存(第二次响应带 Age 头)。 Cloud CDN 不缓存 HEAD——别用 curl -I 测。

第 6 步 — 预签发 TLS 证书

经典 Google 托管证书要等 DNS 指向负载均衡器之后才能激活——对零停机 迁移是"先有鸡还是先有蛋"的问题。用 Certificate Manager DNS 授权 解决:一条一次性 CNAME 证明域名所有权,在 CloudFront 仍承载 100% 流量时就把证书 签出来。把证书 map 挂到 Terraform 创建的 HTTPS 代理上(map 优先级高于经典证书), 然后在动 DNS 之前先验证:

gcloud certificate-manager dns-authorizations create dit-authz --domain=img.example.com
gcloud certificate-manager certificates create dit-cert \
    --domains=img.example.com --dns-authorizations=dit-authz
gcloud compute target-https-proxies update <prefix>-https-proxy --certificate-map=dit-cert-map

curl -sv --resolve img.example.com:443:<lb_ip> https://img.example.com/<test-path> -o /dev/null

第 7 步 — 加权 DNS 切流

两侧证书就绪后,给同一域名建两条加权记录——一条 A 记录指向 GCP 负载均衡器 IP,一条保留指向 CloudFront 的别名——按 10% → 50% → 100% 逐步提升,每一步留观察期,全程 TTL 保持 60 秒。GCP 侧盯这些:

# 负载均衡器:4xx/5xx 比率(和你的 CloudFront 基线比,不是和零比)
gcloud logging read 'resource.type="http_load_balancer" httpRequest.status>=400' --freshness=1h

# Cloud Run:应用错误
gcloud logging read 'resource.type="cloud_run_revision" severity>=ERROR' --freshness=1h

同时关注 Cloud CDN 缓存命中率——从接近 0% 起步,随缓存预热几小时内爬向你的 CloudFront 基线。可选:提升权重前用 top-N URL 对新端点 GET 一遍做预热。灰度期间 让 S3→GCS 传输任务持续按计划运行;到 100% 后,把写入路径(CMS、 媒体流水线)指向 GCS,最后跑一次传输清尾差,再停掉任务。

回滚预案

回滚只是一次 DNS 权重变更:把 CloudFront 拨回权重 100(一个 TTL 内生效—— 60 秒)。迁移全程从未改动 AWS 栈,它直接恢复承载。唯一需要对账的状态是写入路径 切换后写进 GCS 的图片——且仅在已迁移写入路径时才存在。这份能力保留到观察期结束 (通常 100% 流量后 1–2 周),然后下线:删除 CloudFormation 栈,S3 只读保留 30–90 天后清理。

最佳实践清单

  • 第一天就把 DNS TTL 降到 60 秒——切流和回滚都变成秒级操作。
  • 绝不重新生成签名 URL——迁移密钥就够了。
  • Storage Transfer Service 要加调度,不要一次性拷贝——否则拷贝与切流之间的增量就是数据丢失窗口。
  • 用 DNS 授权预签发证书——永远不要让证书签发卡住切流。
  • 首次部署所有参数值与 AWS 保持一致;想改进,等迁移稳定后一次改一个变量。
  • 用生产 URL 验证,不要只跑合成测试。
  • 加权 DNS 灰度;错误率和你的 CloudFront 基线比,不是和零比。
  • 观察期结束前不要下线 AWS——回滚必须始终是一个 60 秒操作。
  • 先迁读路径,最后迁写路径。
