架构概览

组件构成

组件职责
全球外部应用负载均衡器(Application Load Balancer) 单一 anycast IP,挂 Google 托管 SSL 证书。/demo/* 路由到 Demo UI 后端桶,/docs/* 路由到文档后端桶,其余路径全部转发到 image-handler API。
Cloud CDN 在 Google 边缘节点缓存处理后的图片。TTL 遵循源站 Cache-Control。缓存键包含归一化后的 Accept 头,WebP 与非 WebP 客户端各自独立缓存。负缓存(negative caching):4xx 缓存 10 秒,5xx 缓存 600 秒。
Cloud Run dit-image-handler Node.js 22 + sharp(TypeScript)服务,1 vCPU / 1 GiB,min-instances=0。负责解析请求(Default / Thumbor / Custom)、读取源对象、执行编辑管线并附带缓存头返回图片。
Cloud Storage 源图桶 存放原图。只有 SOURCE_BUCKETS 白名单内的桶可读,其余一律返回 403 ImageBucket::CannotAccessBucket。
Secret Manager 以 JSON 文档形式保存签名密钥,SECRET_KEY 指定取哪个键。HMAC-SHA256(hex)算法与 AWS 方案逐字节一致。
Cloud Vision API FACE_DETECTION 支撑 smartCrop;SAFE_SEARCH_DETECTION 支撑 contentModeration。对标 Amazon Rekognition。
Cloud Logging / Monitoring 结构化请求日志(含 request ID)与可选的运维 Dashboard。

CDN 缓存处理结果
  1. 客户端请求。URL 为 base64 编码的 JSON、Thumbor 风格路径或自定义重写模式三者之一(见 API 参考)。
  2. 边缘层。负载均衡器终结 TLS;命中缓存时 Cloud CDN 直接应答,响应中会出现 Age 头。
  3. Cloud Run。未命中的请求经 serverless NEG(网络端点组)到达 image-handler。入站流量(ingress)仅限来自负载均衡器。
  4. 请求归一化(对标 CloudFront Function):Accept 头收敛为 image/webp 或空;query 只保留 signature, expires, format, fit, width, height, rotate, flip, flop, grayscale,随后排序,保证 CDN 缓存键稳定。
  5. 校验。请求类型判定、桶白名单、可选的签名校验(Secret Manager)与 expires 检查。失败时返回与 AWS 一致的错误 JSON。
  6. 取源。从 Cloud Storage 读取对象;对象不存在返回 404 NoSuchKey。
  7. 图片变换。sharp 先执行 resize,其余 edits 按键序执行。smartCrop 与 contentModeration 调用 Vision API。若既无 edits 也无 outputFormat,原始字节直接返回,不经过 sharp。
  8. 响应。Cache-Control 取源对象元数据,缺省为 max-age=31536000,public;Cloud CDN 缓存变换结果,后续请求不再触达 Cloud Run。

请求类型判定(顺序敏感)

  1. Default —— 路径符合 base64 正则且能解码为 JSON:/<base64(JSON)>。
  2. Custom —— 同时配置了 REWRITE_MATCH_PATTERN 与 REWRITE_SUBSTITUTION;重写后的路径再按 Thumbor 解析。
  3. Thumbor —— 路径无扩展名,或以 .jpg .jpeg .png .webp .tiff .tif .svg .gif .avif 结尾。
  4. 其余情况 —— 400 RequestTypeError。

与 AWS 方案的差异

  • 没有 6 MB / 29 秒硬限制。Cloud Run 不存在 Lambda 的响应体积限制和 API Gateway 的超时限制。如需精确复刻 AWS 的 413 TooLargeImageException 行为,设 COMPAT_AWS_LIMITS=Yes。
  • 桶前缀。Thumbor 路径同时接受 s3:<bucket>/(兼容保留)与 gs:<bucket>/;BUCKET_MAP 可把旧 S3 桶名映射到 GCS 桶名。
  • 人脸排序。Vision API 不保证与 Rekognition 相同的返回顺序,本方案统一按人脸框(bounding box)面积降序排序,保证 faceIndex 结果确定。多人脸图片上可能与 AWS 结果不同,属已文档化的差异。
