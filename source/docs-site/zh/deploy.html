部署方案

两种部署方式产出完全相同的基础设施:

  • 方式一 —— Launch Wizard(对标 CloudFormation 一键模板):Cloud Shell 交互式脚本,收集参数后代你驱动 Terraform。
  • 方式二 —— Terraform(对标 CDK 源码部署):直接调用模块,自行管理变量与状态。

前置条件

  • 已开通结算的 Google Cloud 项目。
  • 创建 Cloud Run 服务、负载均衡器、桶、密钥及服务级 IAM 绑定的权限(大致相当于:Cloud Run Admin、Compute Load Balancer Admin、Storage Admin、Secret Manager Admin、Service Usage Admin)。
  • 如需自定义域名,准备一个你可控的域名(部署会签发 Google 托管证书,DNS A 记录由你添加)。
  • 一个或多个已存在的、存放源图的 Cloud Storage 桶。

方式一:Launch Wizard(Open in Cloud Shell)

  1. 在项目中打开 Cloud Shell(或点击仓库 README 中的 Open in Cloud Shell 链接,会自动克隆仓库)。
  2. 若是手动打开的 Cloud Shell,先克隆仓库: 
    git clone https://github.com/leonli/dynamic-image-transformation-gcp.git
cd dynamic-image-transformation-gcp
  3. 运行向导: 
    ./infra/launch-wizard/launch-wizard.sh
  4. 按提示回答问题——问题项与 CloudFormation 参数表一一对应(源桶、CORS、自动 WebP、签名、兜底图、Demo UI、域名、区域)。直接回车即接受默认值。
  5. 向导会先执行 terraform plan 并展示摘要,确认后再实际创建。
  6. 结束时向导会打印负载均衡器 IP、API 端点以及 Demo UI / 文档站地址。为域名添加 DNS A 记录,等待托管证书变为 ACTIVE(DNS 生效后通常需 15–60 分钟)。

向导只是下方 Terraform 模块的轻量前端——换参数重跑即更新同一套部署;--destroy 参数负责拆除(见卸载方案)。

方式二:Terraform

cd infra/terraform
terraform init

cat > terraform.tfvars <<'EOF'
project_id     = "my-project"
region         = "asia-southeast1"
source_buckets = "my-image-bucket,my-other-bucket"
domain         = "img.example.com"
auto_webp      = "Yes"
EOF

terraform plan
terraform apply

输入变量

变量与 AWS 方案的 CloudFormation 参数一一对应(命名调整为 Terraform 习惯)。Yes/No 保持字符串形式,与 AWS 模板对齐。

变量默认值说明
project_id— (必填)目标 Google Cloud 项目。
regionasia-southeast1Cloud Run 与方案自建桶所在区域;负载均衡器与 CDN 为全球资源。
source_buckets— (必填)源图桶白名单,逗号分隔,第一个为默认桶。写入 SOURCE_BUCKETS。
domain""托管 SSL 证书使用的自定义域名(如 img.example.com)。留空则仅以 LB IP 提供 HTTP(评估用)。
deploy_demo_ui"Yes"是否部署 Demo UI 桶及 /demo/* 路由。
cors_enabled"No"是否输出 CORS 头。写入 CORS_ENABLED。
cors_origin"*"启用 CORS 时 Access-Control-Allow-Origin 的取值。写入 CORS_ORIGIN。
auto_webp"No"客户端 Accept 头包含 image/webp 时自动输出 WebP。写入 AUTO_WEBP。
enable_signature"No"是否强制 HMAC-SHA256 URL 签名。写入 ENABLE_SIGNATURE。
secret_name"dit-signature-secret"存放签名 JSON 的 Secret Manager 密钥名。写入 SECRETS_MANAGER。
secret_key""密钥 JSON 中作为 HMAC 密钥的键名。写入 SECRET_KEY。
enable_default_fallback_image"No"处理失败时返回兜底图而非错误体。写入 ENABLE_DEFAULT_FALLBACK_IMAGE。
fallback_image_bucket""兜底图所在桶。写入 DEFAULT_FALLBACK_IMAGE_BUCKET。
fallback_image_key""兜底图对象 key。写入 DEFAULT_FALLBACK_IMAGE_KEY。
rewrite_match_pattern""Custom 请求重写正则。写入 REWRITE_MATCH_PATTERN。
rewrite_substitution""Custom 请求替换串。写入 REWRITE_SUBSTITUTION。
bucket_map""GCP 独有。逗号分隔的 s3名=gcs名 别名映射,让旧 s3:<bucket>/ URL 直接解析到 GCS 桶。写入 BUCKET_MAP。
compat_aws_limits"No"GCP 独有。复刻 Lambda 6 MB 响应限制(413 TooLargeImageException)。写入 COMPAT_AWS_LIMITS。
sharp_size_limit""覆盖 sharp 的 limitInputPixels。写入 SHARP_SIZE_LIMIT。
log_retention_days30请求日志在 Cloud Logging 的保留天数(对标 Lambda 日志保留参数)。

输出

  • lb_ip —— 全球 anycast IP,域名 A 记录指向它。
  • api_url —— 图片 API 基础地址。
  • demo_url / docs_url —— Demo UI 与本指南地址(如部署)。
  • cloud_run_service —— dit-image-handler 服务名。

部署验证

# 1. Plain fetch through the CDN (replace with your domain and bucket/key)
REQ='{"bucket":"my-image-bucket","key":"photos/cat.jpg","edits":{"resize":{"width":200}}}'
curl -o out.jpg "https://img.example.com/$(echo -n "$REQ" | base64 -w0)"

# 2. Second request should be an edge cache hit — look for the Age header
curl -sI "https://img.example.com/$(echo -n "$REQ" | base64 -w0)" | grep -i -e age -e cache-control

仓库还附带 deployment/run-e2e-tests.sh,可对已部署端点断言缩放、格式转换、水印、智能裁剪、签名、兜底图、CORS、自动 WebP 及 CDN 缓存行为。
