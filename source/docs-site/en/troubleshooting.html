Troubleshooting

403 SignatureDoesNotMatch

The computed HMAC differs from ?signature=. Checklist:

  • String-to-sign must be the path including the leading slash, plus ? and the remaining query parameters sorted by key (with signature itself removed). No URL re-encoding in between.
  • JSON byte-identity. The signature covers the base64 string, so any re-serialization of the JSON (key order, whitespace, separators) produces a different path. Sign the exact string you send.
  • Secret value. The secret is a JSON document in Secret Manager; the HMAC key is the value at SECRET_KEY, not the whole JSON. Verify with: 
    gcloud secrets versions access latest --secret=dit-signature-secret
  • Digest format is lowercase hex, not base64.
  • A missing (rather than wrong) signature returns 400 AuthorizationQueryParametersError; a 500 SignatureValidationFailure usually means the service cannot read the secret — check the runtime service account has roles/secretmanager.secretAccessor.

404 NoSuchKey

“The image <key> does not exist or the request may not be base64 encoded properly.” Any GCS read failure maps to this error, so check in order:

  1. Decode your own URL and inspect the JSON: 
    echo 'eyJrZXkiOiJwaG90b3MvY2F0LmpwZyJ9' | base64 -d
    Truncated or URL-mangled base64 (a stripped trailing =, a + converted to a space by an intermediate system) is the most common cause.
  2. Object exists? gsutil ls gs://my-image-bucket/photos/cat.jpg — keys are case-sensitive and must not start with a slash.
  3. Permissions. The Cloud Run runtime service account needs roles/storage.objectViewer on the source bucket. Permission-denied reads also surface as 404 (by design, to avoid leaking bucket topology).
  4. Wrong bucket. If bucket is omitted, the first entry of SOURCE_BUCKETS is used — confirm it is the one you expect.

CDN cache not hitting

Diagnose with response headers:

curl -sI "https://img.example.com/<base64>" | grep -i -e age -e cache-control -e etag
  • Age present and increasing → the response came from the CDN edge cache. No Age → it was a miss (or the object is not cacheable).
  • Check Cache-Control. Cloud CDN honors the origin header. If the source object carries Cache-Control: private or no-store metadata, it is passed through and the CDN will not cache — fix the object metadata (gsutil setmeta -h "Cache-Control:max-age=31536000,public" gs://…) or rely on the default max-age=31536000,public by clearing it.
  • expires URLs rewrite Cache-Control to the remaining lifetime — short-lived links cache poorly by design.
  • Unstable URLs. The cache key is the full URL plus the normalized Accept header. Randomized query parameters are stripped by the normalizer (so they don’t fragment the cache), but differing paths — e.g. JSON with fluctuating key order — create distinct entries. Emit the JSON deterministically.
  • First request after deploy is always a miss; error responses cache only 10 s (4xx) / 600 s (5xx).

Vision-related errors (smartCrop / contentModeration)

  • 500 SmartCrop::Error or 500 Rekognition::DetectModerationLabelsError — the Vision API call failed. Verify the API is enabled (gcloud services enable vision.googleapis.com) and the runtime service account has permission to call it, then check quota (~1,800 requests/min by default).
  • 400 SmartCrop::FaceIndexOutOfRange — fewer faces were detected than faceIndex expects. Remember faces are ordered by bounding-box area (largest first), which can differ from Rekognition ordering for multi-face images.
  • Moderation seems to “do nothing” — the likelihood mapping is coarse (steps of 25); a minConfidence of 75 requires LIKELY or stronger. Lower it to 50 to also catch POSSIBLE.
