Deploy the solution

Two deployment paths produce identical infrastructure:

  • Option 1 — Launch Wizard (counterpart of the CloudFormation one-click template): an interactive Cloud Shell script that collects parameters and drives Terraform for you.
  • Option 2 — Terraform (counterpart of the CDK source deployment): call the module directly with your own variable values and state management.

Prerequisites

  • A Google Cloud project with billing enabled.
  • Permissions to create Cloud Run services, load balancers, buckets, secrets and service IAM bindings (roughly: Cloud Run Admin, Compute Load Balancer Admin, Storage Admin, Secret Manager Admin, Service Usage Admin).
  • A domain you control if you want a custom hostname (a Google-managed certificate is provisioned by the deployment; you create the DNS A record).
  • One or more existing Cloud Storage buckets containing your source images.

Option 1: Launch Wizard (Open in Cloud Shell)

  1. Open Cloud Shell in your project (or click the Open in Cloud Shell link from the repository README, which clones the repo automatically).
  2. Clone the repository if you opened Cloud Shell manually: 
    git clone https://github.com/leonli/dynamic-image-transformation-gcp.git
cd dynamic-image-transformation-gcp
  3. Run the wizard: 
    ./infra/launch-wizard/launch-wizard.sh
  4. Answer the prompts — they mirror the CloudFormation parameter table (source buckets, CORS, auto-WebP, signature, fallback image, demo UI, domain, region). Press Enter to accept defaults.
  5. Review the generated plan (the wizard runs terraform plan and shows a summary) and confirm to apply.
  6. When the wizard finishes it prints the load balancer IP, the API endpoint, and the Demo UI / docs URLs. Create the DNS A record for your domain and wait for the managed certificate to become ACTIVE (typically 15–60 minutes after DNS propagates).

The wizard is a thin front-end over the Terraform module below — re-running it with different answers updates the same deployment, and --destroy tears it down (see Uninstall).

Option 2: Terraform

cd infra/terraform
terraform init

cat > terraform.tfvars <<'EOF'
project_id     = "my-project"
region         = "asia-southeast1"
source_buckets = "my-image-bucket,my-other-bucket"
domain         = "img.example.com"
auto_webp      = "Yes"
EOF

terraform plan
terraform apply

Input variables

The variables map one-to-one to the CloudFormation parameters of the AWS solution (names adapted to Terraform conventions). Values Yes/No are kept as strings for parity with the AWS templates.

VariableDefaultDescription
project_id— (required)Target Google Cloud project.
regionasia-southeast1Region for Cloud Run and solution buckets. The load balancer and CDN are global.
source_buckets— (required)Comma-separated allow list of source image buckets; the first entry is the default bucket. Sets SOURCE_BUCKETS.
domain""Custom hostname for the managed SSL certificate (e.g. img.example.com). Empty = HTTP on the LB IP only (evaluation).
deploy_demo_ui"Yes"Deploy the Demo UI bucket and the /demo/* route.
cors_enabled"No"Emit CORS headers. Sets CORS_ENABLED.
cors_origin"*"Value of Access-Control-Allow-Origin when CORS is enabled. Sets CORS_ORIGIN.
auto_webp"No"Serve WebP automatically when the client’s Accept header includes image/webp. Sets AUTO_WEBP.
enable_signature"No"Require HMAC-SHA256 URL signatures. Sets ENABLE_SIGNATURE.
secret_name"dit-signature-secret"Secret Manager secret holding the signature JSON. Sets SECRETS_MANAGER.
secret_key""Key inside the secret JSON whose value is the HMAC secret. Sets SECRET_KEY.
enable_default_fallback_image"No"Serve a fallback image instead of an error body when processing fails. Sets ENABLE_DEFAULT_FALLBACK_IMAGE.
fallback_image_bucket""Bucket of the fallback image. Sets DEFAULT_FALLBACK_IMAGE_BUCKET.
fallback_image_key""Object key of the fallback image. Sets DEFAULT_FALLBACK_IMAGE_KEY.
rewrite_match_pattern""Regex for Custom request rewriting. Sets REWRITE_MATCH_PATTERN.
rewrite_substitution""Substitution for Custom request rewriting. Sets REWRITE_SUBSTITUTION.
bucket_map""GCP-only. Comma-separated s3name=gcsname aliases so legacy s3:<bucket>/ URLs resolve to GCS buckets. Sets BUCKET_MAP.

sharp_size_limit""Override sharp’s limitInputPixels. Sets SHARP_SIZE_LIMIT.
log_retention_days30Cloud Logging retention for the request logs (counterpart of the Lambda log retention parameter).

Outputs

  • lb_ip — global anycast IP; point your domain’s A record here.
  • api_url — image API base URL.
  • demo_url / docs_url — Demo UI and this guide (when deployed).
  • cloud_run_service — the dit-image-handler service name.

Validate the deployment

# 1. Plain fetch through the CDN (replace with your domain and bucket/key)
REQ='{"bucket":"my-image-bucket","key":"photos/cat.jpg","edits":{"resize":{"width":200}}}'
curl -o out.jpg "https://img.example.com/$(echo -n "$REQ" | base64 -w0)"

# 2. Second request should be an edge cache hit — look for the Age header
curl -sI "https://img.example.com/$(echo -n "$REQ" | base64 -w0)" | grep -i -e age -e cache-control

The repository also ships deployment/run-e2e-tests.sh, which asserts resize, format conversion, watermarking, smart crop, signatures, fallback, CORS, auto-WebP and CDN cache behavior against a deployed endpoint.
