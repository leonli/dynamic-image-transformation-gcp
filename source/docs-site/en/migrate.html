Migrate from AWS

This page is for teams already running the AWS solution Dynamic Image Transformation for Amazon CloudFront (v7, serverless) who want to move to this solution with zero client-side changes. The URL API, the signature algorithm and the error contract are identical to AWS (see the compatibility specification) — your apps, mobile clients, CMS templates and already-issued signed URLs keep working. The only thing that changes is where the DNS record points.

The full, copy-pasteable version of this guide — including the S3 log-replay script and every CLI command — lives in the repository: docs/MIGRATION.md. This page summarizes the plan, the steps and the best practices.

Migration at a glance

Both stacks run in parallel during the whole migration; there is no big-bang moment and no maintenance window. Traffic moves by weighted DNS; images flow S3 → GCS through an incremental, repeatable transfer job.

Step 1 — Assess the AWS deployment

Pull the CloudFormation parameters (aws cloudformation describe-stacks) and the image handler Lambda's environment variables — they map 1:1 onto this solution's Terraform variables. From a day of CloudFront access logs, identify which request types production actually uses: paths starting with base64 (/eyJi…) are Default, /fit-in/… and /filters:… are Thumbor, anything else is Custom.

Do this on day one: lower the DNS TTL of your image hostname to 60 seconds. TTL changes need one full old-TTL period to propagate, so doing it early makes both cutover and rollback instant, for free.

BUCKET_MAP is the key migration helper. If your production URLs embed S3 bucket names (a "bucket" field in Default request JSON, or paths like /my-s3-bucket:photos/cat.jpg), you do not need to re-generate any URLs. Set bucket_map = "my-s3-bucket=my-gcs-bucket" and every request naming the S3 bucket is transparently served from the GCS bucket. Explicit s3: prefixes are accepted and mapped the same way; the SOURCE_BUCKETS allow-list check runs on the mapped name, matching AWS semantics.

Step 3 — Copy the images (S3 → GCS)

Use Storage Transfer Service: managed, parallel, checksummed and — most importantly — incremental and repeatable. Run it once for the bulk copy, then re-run (or schedule) it so late writes to S3 keep flowing in until the write path is cut over:

gcloud transfer jobs create s3://my-s3-bucket gs://my-gcs-bucket \
    --source-creds-file=/tmp/aws-creds.json --name=dit-migration-images
gcloud transfer jobs run dit-migration-images   # incremental re-run, any time

Keys must stay identical. The solution addresses objects by the exact key in the URL/JSON — do not rename, re-prefix or flatten paths during the copy. If ENABLE_DEFAULT_FALLBACK_IMAGE=Yes, copy the fallback image object too.

Step 4 — Copy the signing secret

Signatures are HMAC-SHA256(path[?sorted-query]) hex digests — the same key produces the same signature on both clouds, so migrating the key means every already-issued signed URL (including long-lived links baked into emails and apps) keeps validating:

aws secretsmanager get-secret-value --secret-id <secret> \
    --query SecretString --output text > /tmp/dit-sig.json

terraform apply -var-file=terraform.tfvars \
    -var "signature_secret_json=$(cat /tmp/dit-sig.json)"

Keep secret_key_name equal to the AWS SecretsManagerKeyParameter. Delete the temp file afterwards and never commit it.

Step 5 — Deploy and validate

Deploy with the Launch Wizard or Terraform, copying your AWS parameter values verbatim. Then validate before any traffic moves:

  • Run the shipped suites: deployment/run-unit-tests.sh and BASE_URL=http://<lb_ip> deployment/run-e2e-tests.sh.
  • Replay real production URLs — extract the top-N paths from CloudFront access logs and request each against both stacks, comparing status code, error JSON and Content-Type (these must match exactly). Byte sizes may differ slightly across encoder versions; compare pixels, not bytes. smartCrop crops are equivalent but not byte-identical (Vision vs. Rekognition bounding boxes).
  • Verify a signed URL generated by your existing AWS signing code validates on GCP, and that a tampered one returns 403 SignatureDoesNotMatch.
  • Test CDN caching with two GETs (second response carries Age). Cloud CDN does not cache HEAD — don't test with curl -I.

Step 6 — Pre-issue the TLS certificate

The classic Google-managed certificate only activates after DNS points at the load balancer — a chicken-and-egg problem for zero-downtime migrations. Solve it with Certificate Manager DNS authorization: a one-off CNAME proves domain ownership and the certificate is issued while CloudFront still serves 100% of traffic. Attach the certificate map to the HTTPS proxy created by Terraform (a map takes precedence over the classic certificate), then verify before touching DNS:

gcloud certificate-manager dns-authorizations create dit-authz --domain=img.example.com
gcloud certificate-manager certificates create dit-cert \
    --domains=img.example.com --dns-authorizations=dit-authz
gcloud compute target-https-proxies update <prefix>-https-proxy --certificate-map=dit-cert-map

curl -sv --resolve img.example.com:443:<lb_ip> https://img.example.com/<test-path> -o /dev/null

Step 7 — Cut over with weighted DNS

With certificates live on both sides, create two weighted records for the same hostname — an A record to the GCP load balancer IP and the existing alias to CloudFront — and shift 10% → 50% → 100% with a bake period at each step. Keep TTL at 60 s throughout. Watch on the GCP side:

# Load balancer: 4xx/5xx rate (compare against your CloudFront baseline, not zero)
gcloud logging read 'resource.type="http_load_balancer" httpRequest.status>=400' --freshness=1h

# Cloud Run: application errors
gcloud logging read 'resource.type="cloud_run_revision" severity>=ERROR' --freshness=1h

Also watch the Cloud CDN cache hit ratio — it starts near 0% and climbs toward your CloudFront baseline within hours as the cache warms. Optionally pre-warm by GETting your top-N URLs through the new endpoint before shifting weight. Keep the S3→GCS transfer job running on a schedule during the canary; once at 100%, point the write path (CMS, media pipeline) at GCS, run one final transfer to drain stragglers, and stop the job.

Rollback plan

Rollback is a DNS weight change, nothing more: set CloudFront back to weight 100 (takes effect within one TTL — 60 s). The AWS stack was never modified during the migration, so it simply resumes. The only state to reconcile is images written to GCS after the write-path move, and only if you already moved it. Keep this ability until the bake period ends (typically 1–2 weeks at 100%), then decommission: delete the CloudFormation stack and keep S3 read-only for a 30–90 day retention window.

Best-practices checklist

  • Lower DNS TTL to 60 s on day one — cutover and rollback become instant.
  • Never regenerate signed URLs — migrate the secret instead.
  • Use Storage Transfer Service on a schedule, not a one-shot copy — the delta between copy and cutover is otherwise your data-loss window.
  • Pre-issue the certificate with DNS authorization — never let cert provisioning gate the cutover.
  • Keep every parameter value identical to AWS on first deploy; improve things after the migration bakes, one variable at a time.
  • Validate with production URLs, not just synthetic tests.
  • Canary with weighted DNS; compare error rates against your CloudFront baseline, not against zero.
  • Don't decommission AWS until the bake period ends — rollback must stay a 60-second operation.
  • Move the read path first, the write path last.
