Architecture overview

Components

ComponentRole
Global external Application Load Balancer Single anycast IP with a Google-managed SSL certificate. Routes /demo/* to the Demo UI backend bucket, /docs/* to the documentation backend bucket, and every other path to the image-handler API.
Cloud CDN Caches transformed images at Google edge locations. TTLs follow the origin Cache-Control header. When AUTO_WEBP is enabled the handler emits Vary: Accept, which Cloud CDN honors natively, so WebP and non-WebP clients get separate cache entries (Cloud CDN does not allow Accept in the cache key itself). Negative caching: 4xx for 10 s, 5xx for 600 s.
Cloud Run dit-image-handler Node.js 22 + sharp (TypeScript) service, 1 vCPU / 1 GiB, min-instances=0. Parses the request (Default / Thumbor / Custom), fetches the source object, applies the edit pipeline and returns the image with cache headers.
Cloud Storage source buckets Hold the original images. Only buckets listed in SOURCE_BUCKETS may be read; anything else is rejected with 403 ImageBucket::CannotAccessBucket.
Secret Manager Stores the signature secret as a JSON document; SECRET_KEY selects the JSON key. The HMAC-SHA256 (hex) algorithm is byte-for-byte identical to the AWS solution.
Cloud Vision API FACE_DETECTION for smartCrop; SAFE_SEARCH_DETECTION for contentModeration. Counterpart of Amazon Rekognition.
Cloud Logging / Monitoring Structured request logs (with request IDs) and an optional operations dashboard.

CDN stores the result
  1. Client request. The URL is either a base64-encoded JSON document, a Thumbor-style path, or a custom pattern (see API reference).
  2. Edge. The load balancer terminates TLS; Cloud CDN answers directly on a cache hit. On a hit you will see an Age header in the response.
  3. Cloud Run. Cache misses reach the image-handler service through a serverless NEG (network endpoint group). Ingress is restricted to traffic coming through the load balancer.
  4. Request normalization (counterpart of the CloudFront Function): the Accept header is collapsed to image/webp or empty, and the query string is filtered to signature, expires, format, fit, width, height, rotate, flip, flop, grayscale, then sorted. This keeps the CDN cache key stable.
  5. Validation. Request type detection, bucket allow list, optional signature verification (Secret Manager) and expires checking. Failures return AWS-compatible error JSON.
  6. Source fetch. The object is read from Cloud Storage. A missing object returns 404 NoSuchKey.
  7. Transformation. sharp applies resize first, then the remaining edits in key order. smartCrop and contentModeration call the Vision API. If there are no edits and no output format, the original bytes are returned untouched.
  8. Response. Cache-Control comes from the source object metadata or defaults to max-age=31536000,public; Cloud CDN caches the transformed result so subsequent requests never reach Cloud Run.

Request type detection (order-sensitive)

  1. Default — the path matches the base64 regular expression and decodes to JSON: /<base64(JSON)>.
  2. Custom — both REWRITE_MATCH_PATTERN and REWRITE_SUBSTITUTION are configured; the rewritten path is then parsed as Thumbor.
  3. Thumbor — the path has no file extension, or ends in .jpg .jpeg .png .webp .tiff .tif .svg .gif .avif.
  4. Anything else — 400 RequestTypeError.

Differences from the AWS solution

  • No 6 MB / 29 s hard limits. Cloud Run does not impose Lambda’s response size or API Gateway’s timeout. Set COMPAT_AWS_LIMITS=Yes to reproduce the AWS 413 TooLargeImageException behavior exactly.
  • Bucket prefixes. Thumbor paths accept both s3:<bucket>/ (kept for compatibility) and gs:<bucket>/. BUCKET_MAP can alias old S3 bucket names to GCS bucket names.
  • Face ordering. Vision API does not return faces in Rekognition’s order; faces are sorted by bounding-box area (descending) so faceIndex stays deterministic. Documented behavior, may differ from AWS for multi-face images.
